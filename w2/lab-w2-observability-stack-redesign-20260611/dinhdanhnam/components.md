# Component Decision Table

| Capability | Chosen component / vendor | Why this one | Pain points addressed | What gets worse if we change our mind in six months | Sources / rationale |
|---|---|---|---|---|---|
| Metrics ingestion + storage + query | OpenTelemetry Collector / Grafana Alloy into self-hosted VictoriaMetrics, queried from Grafana | It keeps Prometheus-compatible queries while moving cardinality controls into ingestion before labels become billable storage. | 4, 8 | Replacing it later means rewriting alert rules, SLO dashboards, relabel policies, and six months of PromQL/query-performance tuning. | VictoriaMetrics `vmagent` supports Prometheus-style scraping, relabeling, and remote-write fanout: https://docs.victoriametrics.com/victoriametrics/vmagent/ |
| Logs ingestion + storage + search | Fluent Bit + Alloy into Loki for 14-day hot search, with filtered S3 Parquet archive for 90-day audit | It keeps recent incident logs searchable in Grafana and moves audit history to cheaper object storage instead of paying Splunk hot-search prices for every byte. | 1, 6, 8, 10 | If reversed, security and on-call users must relearn search workflows, LogQL/Splunk saved searches need another migration, and audit report parity must be revalidated. | Loki retention is managed through the compactor/retention model: https://grafana.com/docs/loki/latest/operations/storage/retention/ ; AWS S3 pricing is public: https://aws.amazon.com/s3/pricing/ |
| Distributed tracing | OpenTelemetry SDKs + Alloy tail sampling into Tempo | Tail sampling keeps complete traces for errors and slow requests, which directly addresses the current blind 1% head sample without retaining every normal trace. | 2, 3 | Switching later forces SDK/exporter changes, retunes sampling policies, and invalidates incident training built around Grafana trace links. | OpenTelemetry sampling concepts: https://opentelemetry.io/docs/concepts/sampling/ ; Collector tail-sampling processor: https://github.com/open-telemetry/opentelemetry-collector-contrib/blob/main/processor/tailsamplingprocessor/README.md |
| Alerting rule engine + correlation / grouping | Grafana Alerting + Alertmanager, with a small in-house correlation worker using service graph, fingerprints, and incident tags | It groups downstream symptoms under likely upstream causes and sends one evidence-rich incident instead of many disconnected pages. | 3, 5, 7, 9 | Replacing it later risks missed or duplicate pages because alert rule semantics, inhibition logic, incident tags, and grouping fingerprints rarely map one-to-one. | Tempo service graphs provide service dependency metrics from traces: https://grafana.com/docs/tempo/latest/metrics-from-traces/service_graphs/ ; SLO burn-rate alerting follows Google SRE Workbook guidance: https://sre.google/workbook/alerting-on-slos/ |
| Incident routing + paging | Grafana Cloud IRM, 35 active users, Statuspage retained | IRM collapses paging, schedules, incident timeline, and Grafana evidence links into one response surface at lower seat cost than the current 65-user PagerDuty deployment. | 3, 5, 7, 8, 9 | Moving again later disrupts schedules, escalation policies, mobile notifications, incident-history export, and on-call muscle memory. | Grafana IRM pricing lists `$20 / active IRM user`: https://grafana.com/pricing/ ; product docs: https://grafana.com/products/cloud/irm/ |
| Dashboards + SLO tracking | Self-hosted Grafana HA with VictoriaMetrics, Loki, and Tempo data sources | Grafana becomes the primary human-facing surface for metrics, logs, traces, service graph navigation, dashboards, and burn-rate alerts. | 3, 8 | If changed later, every dashboard, SLO panel, query link, and incident evidence URL embedded in runbooks must be retested. | Google SRE monitoring guidance emphasizes symptom-focused monitoring and SLOs: https://sre.google/sre-book/monitoring-distributed-systems/ and https://sre.google/sre-book/service-level-objectives/ |

## Explicit Mapping To Pain Points

| Pain point | Design response |
|---|---|
| 1. Log search over 7 days is slow | Keep 14 days in Loki for active incidents; push 90-day audit logs to S3 Parquet queried through Athena instead of relying on Splunk saved searches during incidents. |
| 2. 1% trace sampling misses tail latency | Tail-sample 100% of errors and p95+ latency traces, 10% of checkout/payment normal traces, and 1% of low-criticality normal traces. |
| 3. No service-graph correlation | Use Tempo service graph metrics plus Alertmanager grouping and a correlation worker to link symptoms to likely upstream causes. |
| 4. Custom-metric cardinality explosions | Enforce Alloy/vmagent relabel rules that drop `customer_id`, `session_id`, request IDs, and unapproved high-cardinality labels before they enter VictoriaMetrics. |
| 5. PagerDuty alert clusters | Route only grouped incidents from Alertmanager into IRM and inhibit downstream symptoms when an upstream service is already paging. |
| 6. Splunk index rotation breaks dashboards | Remove Splunk from the incident hot path; Loki replication and object-store-backed storage avoid dependency on a vendor hot-index rotation event. |
| 7. No audit trail of incident decisions | Store IRM timeline exports and typed actions in Postgres with service, action, deploy, and rollback tags. |
| 8. Onboarding takes 2-3 weeks | Standardize the first-response workflow on Grafana + IRM and standardize telemetry vocabulary on OTLP/OpenTelemetry semantics. |
| 9. Cannot query incidents by action | Typed incident decision store supports queries like `service=payment-svc action=rollback since=90d`. |
| 10. Vendor lock-in / renewal windows | Retain only operationally valuable SaaS, keep IRM month-to-month where possible, and move core telemetry data to OSS stores plus S3 in the company account. |
